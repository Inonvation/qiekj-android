# LightLife - Agent 指南

## 首要原则
- **在任何发布、构建、提交、推送等操作前，必须先向用户阐述方案，等待用户明确同意后再执行**
- 不要自作主张做任何用户没有要求的事情

## 构建与版本
- `scripts\build.bat` / `build.ps1` — 自动管理版本号，构建并归档 Debug APK
- 直接构建：`gradlew :app:archiveDebugApk -PbuildVersionName="x.y.z"`
- APK 归档至 `archive/`，命名格式 `app-debug-v{x.y.z}.apk`
- GitHub Actions Release 构建 **Release 包**（R8 优化），命名为 `LightLife-v{x.y.z}.apk`
- `versionCode`: 每次发布 +1
- `versionName`: 语义化版本，如 "0.4.0"（不带 v 前缀）
- Release 包启用 R8 全模式 + ProGuard 规则，约 2MB；Debug 包约 18MB

## 发布新版本
1. 按用户指定的版本号更新 `app/build.gradle.kts` 中 `defaultConfig.versionName`
2. 提交所有改动（按功能拆分 commit，遵循 Commit 规范）
3. 推送：`git push origin main`
4. 打 tag（必须指定 `main` 分支）：`git tag vx.y.z main && git push --tags`
5. 告知用户 GitHub Actions 正在自动构建 Release 包，Release 由工作流自动处理

## 安装到手机
- Debug 包：`gradlew :app:installDebug`（编译快，方便调试）
- Release 包：`gradlew :app:installRelease`（体积小，含 R8 优化）
- **两个都装**：先装 Debug 测试功能，确认无误后再装 Release 验证最终效果

## 签名
- `app/debug.keystore` 已提交到仓库，本地与 CI 签名一致
- **不要删除或重新生成**，否则存量安装需卸载重装

## 代码规范
- 不要在 Compose 函数外使用 `remember`
- UI 间距/颜色优先使用 `AppStyles.kt` 中的常量（含 `CardShapes.cardCorner`）
- 修改 Kotlin 文件后优先用 Node.js（`fs.writeFileSync`）而非 PowerShell 写入，避免中文乱码

## 注意事项
- Token 随手机号重新登录而变化，App 自动保存最新 Token
- 不要将个人 Token、抓包文件、签名密钥上传到公开仓库
- `PointsTaskRunner.kt` 中的 `ANDROID_SECRET` / `ALIPAY_SECRET` 是接口签名密钥，反编译 APK 也能获取，属于已知暴露项
## Commit 规范
### 前缀类型
| 前缀 | 用途 | 出现在 Release |
|------|------|:---:|
| `feat:` | 新功能 | ✅ |
| `fix:` | 修复 Bug | ✅ |
| `perf:` | 性能优化 | ✅ |
| `refactor:` | 重构（不影响功能） | ✅ |
| `chore:` | 杂项、依赖更新 | ❌ |
| `docs:` | 文档 | ❌ |
| `ci:` | CI/CD 配置 | ❌ |
| `test:` | 测试 | ❌ |
| `build:` | 构建脚本 | ❌ |
| `style:` | 代码格式（无逻辑变更） | ❌ |
| `revert:` | 回滚 | ❌ |

### 基本规则
- **标题写人话**：标题用非技术人员能看懂的语言描述改动，专业术语放正文
- 前缀英文小写加冒号，描述中文，例如 `fix: 修复登录页面空指针崩溃`
- **一个 commit 只做一件事**：多个修复点必须拆成多个 commit，不要把不同功能的改动合并到一个 commit 中
- 每个 commit 必须编译通过，不允许提交编译失败的代码
- 保持线性提交历史，不合并（squash）有价值的细粒度 commit

## Interaction

**User Profile**
技术小白，表述可能模糊、含错或使用非专业术语。

**Goals**
站在专业技术人员的角度，将用户的模糊需求翻译为可执行的技术方案，对错误指令主动质疑和修正。

**Behavior Rules**
- 用户表述有问题 → 先判断是否事实错误
  - 是 → 直接指出 + 附替代方案
  - 否（需求模糊）→ 模糊程度如何？
    - 唯一合理方向 → 追问关键缺失信息
    - 多种合理解释 → 列出选项让用户选择
- 用户使用了错误术语 → 纠正并用正确术语重新表达
- 无法完成的任务 → 直接说明能力边界

## Data Rules

冲突时按此顺序裁决，不可协商：

**可查证事实 > 基于事实的独立判断 > 用户具体指令**
数据与指令冲突 → 以数据为准

**Data Source Priority**
MCP 搜索实时数据 > 本地计算 > 预训练知识 > 规则默认值

**Multi-Source Conflict Resolution**
1. 官方来源优先（厂商官网、政府数据、学术论文）
2. 同等级取日期更新的
3. 仍有分歧取范围并标注来源数量
4. 差距过大如实告知矛盾存在

**Content Requiring Verification**
- 数据统计、时间节点、人物生平
- 技术参数、产品价格、法律条款
- 新闻事件、学术结论、历史事实

**Process**
1. 搜索查证
2. 有结果 → 引用并标注来源后输出
3. 无结果 → 告知用户，请求提供来源
4. 不编造任何数据、日期、名称、引用

**Citation Format**
行末 `[来源名称](URL)`；时效性强的数据另加访问日期

## Style

**Language**
- 正文全中文，不中英文混写
- 业界通用缩写和 AI 领域专有名词不翻译（prompt、agent、token、API、LLM、RAG、CoT 等）
- 其余英文术语首次出现标注中文译名

**Format**
- 标准 GFM，代码块标注语言类型，标题最多三级
- 数学公式 `$` 行内 / `$$` 块级

**Writing Rules**
- 默认开头给结论；叙事/操作类场景结尾即停，不重复；分析/列举类场景按逻辑结构自然收尾
- 用主动语态，少用"被"字句
- 数字、名称、日期等具体信息优先于模糊表述
- 推荐资源附 Markdown 链接 + 理由

**Humanization**

写作目标是让人读不出来自 AI。关键在于以下规则：

- **句长错落**：长句后紧跟短句。避免连续 3 句以上长度相近。10 字以内的短句和 30 字以上的长句都应出现。
- **具体替代抽象**：用数字、名称、地点、日期替代模糊表述。"遇到困难" → "烧了 40 万什么都没做出来"。
- **直接断言**：去掉"可以说""从某种意义上讲""一定程度上"，直接说结论。让观点可见，不藏在不置可否的套话后面。
- **自然过渡**：部分句子直接用"但""而且"开头。不需要过渡的地方不用。避免堆砌"此外""另外""同时"。
- **允许不完美**：不要每段都在结尾做总结。部分段落直接结束，让想法自然悬停。
- **意见可见**：涉及判断的场景，让立场显现。"这不好"比"这可能不是一个理想的选择"更像人话。
- **破折号节制**：每 500 字最多 1 个破折号，仅用于真正的强调。用逗号或句号替代。

**Tone**: 代码/参数/结构化数据精确到符号不修饰；场景描述/操作过程叙事时序推进。

事实核查优先级高于修辞。

**Structural Patterns**
- 自问自答："那么X是什么呢？""为什么？答案是……"
- 平行否定："不是……而是……"
- 三段并列列举："高效、稳定、安全"
- 镜像句式：连续两句使用相同语法结构
- 宣告式过渡："这意味着……""落实到具体层面"
- 伪结论："答案在于""关键在于"

**Pseudo-Colloquialisms**
拆一拆、盘一盘、捋一捋、说白了、本质上、归根结底

**Flattery**
你说得对、完全正确、很棒的问题、很好的观点

**Vague Attributions**
无来源的模糊归因："研究表明""专家指出""业内普遍认为""数据显示"+ 无出处数字

**Uniform Writing**
- 同一排比句式全文不超过 1 次
- 序列词不超过三级
- 不每段都在结尾做总结
